# Тест-кейсы для API (Задание 2.1)

В этом файле описаны тест-кейсы для проверки 4-х эндпоинтов микросервиса объявлений (https://qa-internship.avito.com).

## 1. POST `/api/1/item` — Создание объявления

### POST-01: Создание валидного объявления со всеми полями

| Поле | Значение |
| :--- | :--- |
| **ID** | POST-01 |
| **Название** | Создание валидного объявления со всеми полями |
| **Приоритет** | High |
| **Цель** | Проверить создание объявления с корректным заполнением всех полей |
| **Тестовые данные** | `{<br> "sellerID": 112233,<br> "name": "Test Item",<br> "price": 500,<br> "statistics": {<br> "likes": 1,<br> "viewCount": 10,<br> "contacts": 2<br> }<br>}` |
| **Шаги** | 1. Подготовить тело запроса с валидными данными<br>2. Отправить `POST /api/1/item` |
| **Ожидаемый результат** | **200 OK**<br>Возвращается `id` созданного объявления (строка или UUID) |

## 2. GET `/api/1/item/:id` — Получение объявления по ID

### GET_ID-01: Получение существующего объявления

| Поле | Значение |
| :--- | :--- |
| **ID** | GET_ID-01 |
| **Название** | Получение существующего объявления по валидному id |
| **Приоритет** | High |
| **Цель** | Проверить получение существующего объявления |
| **Предусловия** | Объявление создано, ID известен |
| **Шаги** | 1. Отправить `GET /api/1/item/{id}` |
| **Ожидаемый результат** | **200 OK**<br>Возвращается массив из одного элемента со всеми полями (`id`, `name`, `price`, `sellerId`, `statistics`) |

### GET_ID-02: Запрос по несуществующему ID

| Поле | Значение |
| :--- | :--- |
| **ID** | GET_ID-02 |
| **Название** | Получение по несуществующему ID |
| **Приоритет** | Medium |
| **Цель** | Проверить обработку запроса с отсутствующим в базе UUID |
| **Тестовые данные** | Рандомный сгенерированный UUID, не существующий в базе |
| **Шаги** | 1. Отправить `GET /api/1/item/{uuid}` |
| **Ожидаемый результат** | **404 Not Found** |

### GET_ID-03: Некорректный формат ID

| Поле | Значение |
| :--- | :--- |
| **ID** | GET_ID-03 |
| **Название** | Передача некорректного формата ID |
| **Приоритет** | Low |
| **Цель** | Проверить валидацию формата ID (не UUID) |
| **Тестовые данные** | Некорректная строка, например `12345` |

## 5. E2E (Сквозные сценарии)

### E2E-01: Создание и получение объявления

| Поле | Значение |
| :--- | :--- |
| **ID** | E2E-01 |
| **Название** | Создание и получение объявления по ID |
| **Приоритет** | High |
| **Цель** | Проверить, что созданное объявление корректно сохраняется и выдается по ID |
| **Шаги** | 1. `POST /api/1/item` (Создать объявление и получить `id`)<br>2. `GET /api/1/item/{id}` (Получить объявление по `id`) |
| **Ожидаемый результат** | Данные из шага 2 (`name`, `price`, `sellerId`) полностью совпадают с отправленными на шаге 1 |

### E2E-02: Получение объявлений продавца после создания

| Поле | Значение |
| :--- | :--- |
| **ID** | E2E-02 |
| **Название** | Создание и проверка в списке продавца |
| **Приоритет** | High |
| **Цель** | Проверить, что новое объявление появляется в списке объявлений продавца |
| **Шаги** | 1. `POST /api/1/item` (Создать объявление для уникального `sellerId` и получить `id`)<br>2. `GET /api/1/{sellerId}/item` |
| **Ожидаемый результат** | Сгенерированный `id` из шага 1 присутствует в списке из шага 2 |
